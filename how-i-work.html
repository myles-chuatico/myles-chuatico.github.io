How I Think

A few things that are
consistently true about me.

01
I build, I don't just document.
When I find a gap, my instinct is to close it. The eLinq pipeline, the mortgage decision tool, the MPR automation suite. None of those were assigned to me. I saw something that wasn't working and built the fix.
02
I learn what the problem needs.
I've written batch scripts, built Power Query pipelines, and shipped a full JavaScript tool. I'm not a coder. I don't let not knowing something stop me. I pick up what the job needs.
03
I care about accuracy.
I don't claim things I didn't do. If a metric is real but unverifiable, I'll say so. If part of a project was someone else's work, I'll tell you. I'd rather undersell and be accurate than oversell and get caught in an interview.
04
Process discipline is non-negotiable.
Everything goes through UAT before production. I use checklists and screenshots because I've seen what happens when you skip it. Proper testing prevents incidents.
05
Give me a problem, not a prescription.
Tell me what's broken. I'll figure out how to fix it. I work best with ownership of something end-to-end: from understanding the gap, through building the solution, to watching it stabilize in production. I don't need to be handed a method.

My Process

How a typical initiative
goes for me.

1
Notice the gap
Something is slow, broken, or invisible. A process that depends on someone remembering to do it manually, or a question no one can answer because the data doesn't exist.
2
Actually understand it
I talk to the people doing the work. I want to know what happens, not what the SOP says happens. Then I check with whoever owns the outcome to align on what fixed looks like.
3
Build something that works
Start with what I know and pick up what I don't. Get a working version in UAT. It doesn't need to be perfect. It needs to work and be testable.
4
Test it properly
UAT is not a checkbox. I test from the user's perspective, including edge cases. Screenshots document everything. Stakeholder sign-off before production.
5
Ship it and watch it
Rollout follows the checklist. I monitor closely for the first few days. If something breaks, I want to catch it first.
6
Document and hand off clean
Work instructions, agent guides, KB articles. If I built it, I document it well enough that someone else can run it without me. The goal is to never be the single point of failure.
